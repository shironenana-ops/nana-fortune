# 音声アップロード鑑定とTTS読み上げ音声生成の責務境界

## 1. このドキュメントの目的

白音七には、既存の音声アップロード鑑定がある。

今後、鑑定本文を読み上げるTTS音声生成を追加する予定がある。

両方とも「音声」に関係するが、責務は違う。このドキュメントは、DB項目、API名、画面表示、課金説明、運用の混同を防ぐための境界線を定義する。

このPRでは実装しない。コード変更、API追加、DB変更、Lambda修正、フロント修正、音声生成処理への接続は行わない。

このドキュメントは公開リポジトリに置く前提で書く。秘密情報、環境固有情報、具体的な本番設定値、認証方式の細部、実際のS3バケット名や詳細パスは記載しない。

## 2. 既存機能: ユーザー音声アップロード鑑定

### 呼称

開発上の呼称候補:

- `voice_upload`
- `uploaded_voice_consultation`
- `voice_consultation`

ユーザー向け表示候補:

- 音声相談
- 声で相談
- 音声相談枠

### 責務

既存の音声アップロード鑑定は、ユーザーが声で相談内容を送るための機能である。

主な責務:

- ユーザーが音声ファイルをアップロードする
- アップロード元音声を保存する
- 文字起こしを行う
- 文字起こし内容をもとに鑑定文を生成する
- 鑑定結果をhistoryへ保存する
- 既存の月額音声相談枠または追加音声相談枠を消費する

### 既存コード上の主な関係ファイル

- `lambda/voice_upload.py`
- `lambda/lambda_function.py`
- `src/pages/premium/voice.astro`
- `src/pages/premium/voice-processing.astro`
- `src/pages/history/[id].astro`
- `src/pages/members.astro`

### 注意

既存処理は、既存の音声アップロード鑑定として扱う。

新規TTS読み上げ音声生成の実装先として流用しない。

既存処理を否定するのではなく、責務が違うものとして分離する。

## 3. 新規予定機能: 鑑定本文のTTS読み上げ音声生成

### 呼称

開発上の呼称候補:

- `tts_audio`
- `reading_audio`
- `fortune_audio`
- `generated_audio`
- `tts_voice`

ユーザー向け表示候補:

- 音声読み上げ
- 声で聴く
- 音声読み上げ枠

### 責務

TTS読み上げ音声生成は、すでに生成された鑑定本文を声で聴けるようにするための機能である。

主な責務:

- すでに生成された鑑定本文をTTSで読み上げる
- 推定秒数から必要な音声読み上げ枠を見積もる
- 生成済みTTS音声をS3等に保存する
- 保存済み音声を再生できるようにする
- 将来的に reserve / commit / release で枠消費を安全管理する
- 同一本文・同一設定の生成済み音声がある場合は再生成せず再利用する
- 再利用時は `charged_slots = 0` として扱う

この機能は、既存の `voice_upload.py` / `lambda_function.py` を流用しない。

## 4. 既存DB項目・history項目の意味

### 既存 `users` 項目

`monthly_voice_limit`:

- 既存の音声アップロード鑑定の月額枠を表す
- 現時点ではTTS読み上げ音声枠に流用しない

`monthly_voice_used`:

- 既存の音声アップロード鑑定の使用数を表す
- 現時点ではTTS読み上げ音声枠の消費には流用しない

`extra_voice_remaining`:

- 既存の追加音声相談枠を表す
- 現時点ではTTS読み上げ音声枠に流用しない

### 既存 `history` 項目

`type = "voice"`:

- 既存の音声アップロード鑑定を指す
- TTS読み上げ音声生成を指す値として流用しない

`source = "premium_voice"`:

- 既存のプレミアム音声アップロード鑑定を指す
- TTS読み上げ音声生成を指す値として流用しない

`audio_s3_key`:

- 既存ではアップロードされた元音声ファイルのS3 keyを指す
- TTS生成済み音声ファイルのS3 keyとして流用しない

`transcript_s3_key`:

- 既存の文字起こし結果を指す

`result_s3_key`:

- 既存の鑑定結果JSONを指す

### TTS読み上げ音声用の項目名候補

TTS読み上げ音声用には、以下のような別項目名を検討する。

- `generated_audio_s3_key`
- `tts_audio_s3_key`
- `reading_audio_s3_key`
- `voice_request_id`
- `source_voice_request_id`

## 5. `UsageType = "voice"` の扱い

PR #68で追加した `UsageType = "voice"` は、現時点では新規TTS読み上げ音声枠の意味として扱う。

既存の音声アップロード鑑定側の `history.type = "voice"` とは同名だが、責務が違う。

以下を方針とする。

- 既存の `lambda/voice_upload.py` / `lambda/lambda_function.py` の処理を `UsageType = "voice"` の新設枠管理へ直接接続しない
- 既存アップロード音声鑑定と新規TTS読み上げ音声生成を同じ枠消費処理に混ぜない
- 将来的に既存アップロード音声鑑定も統合する場合は、別PRで移行設計を立てる

混同防止のため、後続PRで必要があれば以下のような命名変更を検討できる。

- `usage_type = "tts_voice"`
- `usage_type = "tts_audio"`
- `usage_type = "reading_audio"`

ただし、このPRではPR #68で追加した型定義を変更しない。今回のPRでは、意味とリスクをドキュメント上で明確化するだけにする。

## 6. 画面上の呼称ルール

ユーザーが混乱しないように、フロント表示上の呼称を分ける。

### 既存音声アップロード鑑定

推奨表示:

- 音声相談
- 声で相談
- 音声相談枠

避けたい表示:

- 音声枠 だけ
- 音声読み上げ枠
- 声で聴く

### 新規TTS読み上げ音声生成

推奨表示:

- 音声読み上げ
- 声で聴く
- 音声読み上げ枠

避けたい表示:

- 音声相談
- 声で相談
- `premium_voice`

### 説明方針

- 「音声相談」は、ユーザーが声で相談する機能
- 「音声読み上げ」は、鑑定結果を声で聴く機能
- 購入画面、会員ページ、履歴詳細、MOSH、特商法、利用規約でも呼称を揃える
- `members.astro` などで既存 `monthly_voice_*` を表示する場合、新規TTS枠と同じ欄に混ぜない
- ユーザー向けには「音声」という単語だけで表示せず、何をする音声なのか分かる表現にする

## 7. 保存期間・聞き直し・コスト防衛

TTS読み上げ音声生成では、以下のコストリスクがある。

- S3保存費
- S3 GET回数
- CloudFront転送量
- 再生回数増加による通信費
- 長期保存によるストレージ増加
- 無制限再生とファイル永続保持の誤解

方針:

- 生成済み音声の聞き直しは、ユーザー体験上は無制限を目指す
- ただし、生成済み音声ファイルの永続保持を無条件に保証するものではない
- `generated_audio_s3_key` / `tts_audio_s3_key` には保持期間やライフサイクル設計が必要
- 保存期間は後続PRまたは規約整備時に決定する
- 特商法、利用規約、MOSH説明では「聞き直し」と「ファイル永続保証」を混同させない
- 削除・退会・期限切れ時のTTS音声ファイルの扱いは別途決定する
- 将来的にはS3ライフサイクルルール、CloudFrontキャッシュ、署名付きURL、有効期限などの設計が必要になる可能性がある

保存期間の候補:

- 生成から30日間保存
- 会員期間中保存
- 履歴保持期間に連動
- ユーザーが保存済み履歴として閲覧できる期間に連動

## 8. API namespace 方針

TTS読み上げ音声生成のAPI名は、既存音声アップロード鑑定と混ざらないようにする。

避けたいnamespace:

- `/voice/estimate`
- `/voice/generate`

理由:

- 既存の `voice_upload.py` や `history.type = "voice"` と混同しやすい
- ユーザー音声アップロード鑑定とTTS読み上げ音声生成の責務が混ざる

候補namespace:

- `/tts/estimate`
- `/tts/generate`
- `/reading-audio/estimate`
- `/reading-audio/generate`
- `/fortune-audio/estimate`
- `/fortune-audio/generate`

推奨:

- 内部実装では `tts` または `reading_audio` を優先する
- ユーザー向け表示では `音声読み上げ` または `声で聴く` を使う
- 後続PRで正式namespaceを決める
- 既存の音声アップロード鑑定APIとは別namespace・別責務にする

## 9. 既存Lambdaに触らない方針

当面触らないファイル:

- `lambda/voice_upload.py`
- `lambda/lambda_function.py`

理由:

- 既存の音声アップロード鑑定の中核である
- 既存文字起こし前提の処理であり、TTS生成処理とは方向が逆
- 既存のquota消費は完了後直更新であり、PR #67のreserve方式とは異なる
- ここへTTS処理を足すと責務が混ざる
- 本番の履歴・課金・音声相談枠に影響する可能性がある

後続でTTSを実装する場合は、新規ファイル・新規API・新規責務として追加する。

## 10. 外部AI・外部資料向けLite版方針

このドキュメントの内容を外部AIや外部協力者へ渡す場合は、情報を抽象化する。

外部共有時に残してよい情報:

- 既存音声アップロード鑑定と新規TTS読み上げ音声生成は責務が違う
- 既存音声アップロード処理とTTS生成処理は混ぜない
- 既存のアップロード音声用項目とTTS生成済み音声用項目は分ける
- 新規TTSは別namespace・別APIで作る
- PR #69では責務整理ドキュメントのみを追加する

外部共有時に抽象化すべき情報:

- 認証方式の詳細
- 秘密鍵やtokenに関する情報
- 実際のS3バケット名や詳細パス
- 本番AWSイベント設定
- quota消費ロジックの細部
- 決済・課金まわりの未公開内部仕様
- 本番環境の具体的設定

外部AI向けには、以下のような短い起動文に圧縮する。

```text
既存の音声アップロード鑑定と、新規TTS読み上げ音声生成は責務が異なる。
既存音声アップロード処理や既存の音声S3項目をTTS生成に流用しない。
PR #69では、まず音声系責務整理ドキュメントを追加する。
具体的な認証情報、S3構造、本番環境設定、課金内部仕様は共有しない。
```
